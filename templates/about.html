SOLID Principles in HouseMate

This project demonstrates all 5 SOLID principles through a working student housing management application.

S Single Responsibility Principle

Definition: A class should have only one reason to change.

Implementation in HouseMate:

  • BillRepository - Database operations ONLY
  • BillValidator - Validation logic ONLY
  • BillService - Business logic coordination ONLY
  • BillController - HTTP request handling ONLY
Example: If we need to change the database from SQLite to PostgreSQL, we only modify BillRepository, not the service or controller.

O Open/Closed Principle

Definition: Software entities should be open for extension but closed for modification.

Implementation in HouseMate:

Cost distribution uses the Strategy Pattern:

  • EqualDistributionStrategy - Split costs equally
  • PercentageDistributionStrategy - Split by percentages
  • FixedAmountDistributionStrategy - Fixed amounts per person
Example: Adding a new strategy (e.g., weighted by income) requires ZERO changes to existing code. Just create a new class implementing ICostDistributionStrategy.

L Liskov Substitution Principle

Definition: Subtypes must be substitutable for their base types.

Implementation in HouseMate:

All cost distribution strategies are fully substitutable:

  • Any strategy can be swapped at runtime
  • BillService works identically with any strategy
  • No special handling needed for different strategies
Example: The system can switch from equal distribution to percentage distribution without any code changes - just pass a different strategy instance.

I Interface Segregation Principle

Definition: Clients should not be forced to depend on interfaces they don't use.

Implementation in HouseMate:

Multiple small, focused interfaces instead of one large interface:

  • IRepository - Only CRUD operations
  • IValidator - Only validation methods
  • INotificationSender - Only notification methods
  • ICostCalculator - Only calculation methods
Example: A class that only needs to validate data depends on IValidator (2 methods), not a "fat" interface with 20+ methods.

D Dependency Inversion Principle

Definition: High-level modules should not depend on low-level modules. Both should depend on abstractions.

Implementation in HouseMate:

BillService depends on interfaces, not concrete classes:

  • Depends on IRepository, not BillRepository
  • Depends on IValidator, not BillValidator
  • Dependencies injected via constructor
Example: We can easily swap BillRepository for MockRepository during testing, without changing BillService code.
